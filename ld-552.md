## 项目主要需求

### 项目概述

TalentFlow 是一个招聘人才管理系统，面向企业 HR、面试官、招聘经理，支持从职位发布、简历投递、面试安排到 Offer 发放的完整招聘流程管理。系统通过看板、日历、时间线等多维度视图，帮助团队高效协作完成招聘任务。

系统采用前后端分离架构：前端 React 18 + TypeScript + Ant Design 5，后端 NestJS + Prisma ORM，数据库 PostgreSQL，JWT 进行身份认证。

### 核心实体（≥ 4个）

#### 1. 职位 Job

```typescript
// Prisma Schema
model Job {
  id              Int       @id @default(autoincrement())
  title           String                    // 职位名称
  department      String                    // 所属部门
  location        String                    // 工作地点
  salaryRange     String                    // 薪资范围（如 "15k-25k"）
  description     String    @db.Text        // 职位描述
  requirements    String    @db.Text        // 任职要求
  headcount       Int                       // 招聘人数
  status          JobStatus                 // DRAFT / OPEN / CLOSED / ARCHIVED
  hiringManagerId Int                       // 招聘经理ID（外键 → User）
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt
  resumes         Resume[]
  offers          Offer[]
}
```

- **业务动作**：
  - 发布职位（DRAFT → OPEN，需填写完整信息）
  - 暂停招聘（OPEN → PAUSED，不接收新简历）
  - 关闭职位（OPEN/PAUSED → CLOSED，停止招聘）
  - 重新打开（CLOSED → OPEN）
  - 归档职位（CLOSED → ARCHIVED）

#### 2. 候选人 Candidate + 简历 Resume

```typescript
model Candidate {
  id        Int       @id @default(autoincrement())
  name      String                          // 姓名
  email     String    @unique               // 邮箱（全局唯一）
  phone     String?                         // 手机号
  source    String                          // 来源（官网/内推/猎头/招聘平台）
  resumes   Resume[]
  offers    Offer[]
}

model Resume {
  id          Int           @id @default(autoincrement())
  candidateId Int                           // 候选人ID（外键 → Candidate）
  jobId       Int                           // 职位ID（外键 → Job）
  resumeUrl   String                        // 简历文件URL
  coverLetter String?       @db.Text        // 求职信
  status      ResumeStatus                  // 状态（见枚举）
  submittedAt DateTime      @default(now())
  updatedAt   DateTime      @updatedAt
  interviews  Interview[]
}
```

- **关系**：一个候选人可以投递多份简历（不同职位），一份简历关联一个职位
- **ResumeStatus 状态流转**：
  ```
  SUBMITTED → SCREENING → SHORTLISTED → INTERVIEWING → OFFERED → HIRED
                    ↘ REJECTED（任何阶段可流转到此）
  ```
- **业务动作**：
  - 投递简历（创建 Resume，状态 SUBMITTED）
  - HR 筛选简历（SUBMITTED → SCREENING → SHORTLISTED / REJECTED）
  - 标记进入面试（SHORTLISTED → INTERVIEWING）

#### 3. 面试 Interview

```typescript
model Interview {
  id            Int             @id @default(autoincrement())
  resumeId      Int                             // 关联简历（外键 → Resume）
  interviewerId Int                             // 面试官ID（外键 → User）
  round         Int                             // 第几轮面试（1, 2, 3...）
  scheduledAt   DateTime                        // 预约时间
  duration      Int                             // 面试时长（分钟）
  type          InterviewType                   // PHONE / ONSITE / VIDEO / TECHNICAL
  feedback      String?         @db.Text        // 面试反馈
  score         Int?                            // 评分（1-10）
  result        InterviewResult                 // PASS / FAIL / PENDING
  notes         String?         @db.Text        // 备注
  createdAt     DateTime        @default(now())
  updatedAt     DateTime        @updatedAt
}
```

- **关系**：一份简历可以有多轮面试（一对多），每轮面试指派一个面试官
- **业务动作**：
  - 安排面试（创建 Interview，result=PENDING，自动更新 Resume 状态为 INTERVIEWING）
  - 完成面试（填写 feedback、score、result）
  - 面试反馈（面试官填写详细评语）
  - 所有面试通过后，可流转到 Offer 阶段

#### 4. Offer

```typescript
model Offer {
  id          Int         @id @default(autoincrement())
  candidateId Int                         // 候选人ID（外键 → Candidate）
  jobId       Int                         // 职位ID（外键 → Job）
  salary      Decimal                     // 薪资
  startDate   DateTime                    // 入职日期
  status      OfferStatus                 // DRAFT / APPROVED / SENT / ACCEPTED / REJECTED / WITHDRAWN
  approverId  Int                         // 审批人ID（外键 → User）
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
}
```

- **OfferStatus 状态流转**：
  ```
  DRAFT → APPROVED → SENT → ACCEPTED
                  ↘ REJECTED
            ↘ WITHDRAWN（SENT 后可撤回）
  ```
- **业务动作**：
  - 创建 Offer（HR 创建草稿，状态 DRAFT）
  - 审批 Offer（招聘经理审批，DRAFT → APPROVED）
  - 发送 Offer（APPROVED → SENT，通知候选人）
  - 候选人确认（SENT → ACCEPTED / REJECTED）
  - 撤回 Offer（SENT → WITHDRAWN）

### 核心页面（≥ 5个）

#### 1. `/jobs` 职位管理页

- **功能**：展示所有职位列表，支持按状态（JobStatus）、部门筛选；支持创建新职位、编辑职位信息；每个职位卡片显示已投递简历数量和招聘进度
- **布局**：顶部筛选栏 + 职位表格/卡片视图切换 + 分页
- **消费实体**：Job
- **API**：`GET /api/jobs`、`POST /api/jobs`、`PATCH /api/jobs/:id`、`PATCH /api/jobs/:id/status`

#### 2. `/jobs/:id` 职位详情页

- **功能**：展示职位完整信息 + 该职位下所有候选人列表 + 面试进度看板
- **布局**：上方职位信息卡片 + 下方 Tab 切换（候选人列表 / PipelineKanban 看板 / 面试日历）
- **消费实体**：Job、Resume、Interview
- **业务逻辑**：
  - 看板视图（PipelineKanban）：按 ResumeStatus 分列展示候选人卡片，支持拖拽流转状态
  - 候选人列表：使用 CandidateCard 组件展示，点击跳转候选人详情
- **API**：`GET /api/jobs/:id`、`GET /api/jobs/:id/resumes`、`GET /api/jobs/:id/interviews`

#### 3. `/candidates` 候选人列表页

- **功能**：展示所有候选人，支持按投递状态（ResumeStatus）、来源（source）、职位筛选；支持搜索（姓名/邮箱）
- **布局**：左侧筛选面板 + 右侧候选人卡片列表（CandidateCard 组件）+ 简历预览抽屉
- **消费实体**：Candidate、Resume
- **API**：`GET /api/candidates?status=&source=&jobId=&keyword=`

#### 4. `/candidates/:id` 候选人详情页

- **功能**：展示候选人完整信息，包括基本信息、所有投递记录、面试记录时间线、Offer 状态
- **布局**：左侧候选人信息卡 + 右侧 Tab（投递记录 / InterviewTimeline 面试时间线 / Offer 状态）
- **消费实体**：Candidate、Resume、Interview、Offer（全部实体）
- **API**：`GET /api/candidates/:id`、`GET /api/candidates/:id/resumes`、`GET /api/candidates/:id/interviews`、`GET /api/candidates/:id/offers`

#### 5. `/interviews` 面试日历页

- **功能**：以日历视图展示所有面试安排，支持月/周/日切换；显示待安排面试列表；支持安排新面试、修改面试时间、完成面试反馈
- **布局**：左侧日历（FullCalendar 或 Ant Design Calendar）+ 右侧面试详情面板 + 待安排列表
- **消费实体**：Interview、Resume
- **API**：`GET /api/interviews?startDate=&endDate=&interviewerId=`、`POST /api/interviews`、`PATCH /api/interviews/:id`

### 横切关注点

#### 1. 认证授权 + RBAC

四种角色：**HR**、**Interviewer**（面试官）、**HiringManager**（招聘经理）、**Admin**

| 角色 | 职位管理 | 简历管理 | 面试管理 | Offer管理 | 数据范围 |
|---|---|---|---|---|---|
| HR | CRUD | CRUD | 安排/查看 | 创建/发送 | 全部数据 |
| Interviewer | 只读 | 只读分配的 | 查看/填写反馈 | 无 | 仅分配给自己的面试 |
| HiringManager | CRUD | 查看 | 查看 | 审批 | 本部门职位数据 |
| Admin | 全部 | 全部 | 全部 | 全部 | 无限制 |

- **后端**：
  - JWT 认证守卫（NestJS Guard）
  - RBAC 角色装饰器：`@Roles(UserRole.HR, UserRole.ADMIN)`
  - 数据范围过滤：面试官只能看到分配给自己的面试（通过 interceptor 实现）
- **前端**：
  - 路由守卫：根据角色控制页面访问
  - 按钮级权限：根据角色显示/隐藏操作按钮
  - 菜单权限：根据角色显示不同侧边栏菜单
- **触达范围**：DB Users 表 → 后端 auth guard → 后端 RBAC decorator → 后端数据范围 interceptor → 前端路由守卫 → 前端菜单/按钮权限 → 前端 store

#### 2. 操作日志 + 状态流转审计

- **审计中间件**（NestJS Interceptor）：记录每次状态变更的完整信息
- **审计记录字段**：操作人ID、操作类型、目标实体（Resume/Interview/Offer）、目标ID、变更前状态、变更后状态、变更原因（可选）、操作时间、IP 地址
- **必须审计的状态流转**：
  - 简历状态变更（ResumeStatus 每次流转）
  - 面试结果记录（InterviewResult 变更）
  - Offer 状态变更（OfferStatus 每次流转）
  - 职位状态变更（JobStatus 变更）
- **存储**：`audit_logs` 数据库表
- **查询**：管理员可在候选人详情页查看该候选人的完整状态流转历史

### 全局异常处理

#### 后端统一异常过滤器

`backend/src/common/filters/http-exception.filter.ts`：
- NestJS ExceptionFilter 实现
- 捕获所有 HttpException 和未处理异常
- 统一返回格式：`{ statusCode, message, error, timestamp, path }`
- 业务异常定义在 `backend/src/common/exceptions/` 目录下（如 `JobNotFoundException`、`InvalidStatusTransitionException`）
- 使用 NestJS Logger 记录异常日志

#### 前端请求拦截器统一错误提示

`frontend/src/utils/request.ts`：
- Axios 响应拦截器：
  - 401：清除 token，跳转登录页
  - 403：提示"您没有权限执行此操作"
  - 404：提示"请求的资源不存在"
  - 422：提示表单校验错误详情
  - 500：提示"服务器内部错误"
- 使用 Ant Design 的 `message.error` 组件展示错误信息
- 网络异常：统一捕获并提示

### 操作日志

通过审计拦截器（NestJS Interceptor）记录关键写操作：

`backend/src/interceptors/audit-log.interceptor.ts`：
- **记录字段**：操作人（userId）、操作类型、目标实体（entityType）、目标ID（entityId）、变更前状态（oldValue）、变更后状态（newValue）、操作时间、IP 地址、User-Agent
- **必须记录的操作**：
  - 职位创建/状态变更
  - 简历投递/状态流转
  - 面试安排/结果记录
  - Offer 创建/审批/发送/状态变更
  - 用户角色变更
- **存储**：`audit_logs` 数据库表（通过 Prisma 管理）
- **查询接口**：`GET /api/audit-logs?entityType=&entityId=&startDate=&endDate=`

### 严禁合并职责到单一文件

以下规则必须严格遵守：

1. **module / controller / service 分离**：NestJS 每个模块必须有独立的 `.module.ts`、`.controller.ts`、`.service.ts` 文件，controller 不得包含业务逻辑
2. **DTO 与 entity 分离**：Prisma schema 中的 model 是数据库实体，API 层使用独立的 DTO 类（`dto/` 目录），两者不得合并
3. **store 与类型分离**：前端每个 store 引用独立的类型定义文件，不得在 store 中内联类型
4. **枚举集中管理**：后端 `constants/enums.ts` 和前端 `constants/enums.ts` 各维护一份，其他文件 import 使用，不得在任何其他位置重复定义
5. **守卫、拦截器、过滤器各自独立**：`guards/`、`interceptors/`、`filters/` 目录下的文件各司其职，不得合并
6. **共用组件独立**：`CandidateCard`、`PipelineKanban`、`InterviewTimeline` 必须独立为文件，不得内联在页面中
7. **业务异常类独立**：`common/exceptions/` 下按业务域分文件，不得全部堆在一个文件

### 核心实体贯穿全栈

每个核心实体必须完整贯穿以下层级：

```
DB Schema (prisma/schema.prisma)
  → Service (modules/*/service.ts)
    → Controller (modules/*/controller.ts)
      → DTO (modules/*/dto/)
        → API (HTTP endpoint)
          → Store (stores/*.ts)
            → Component (pages/**/*.tsx)
```

以简历 Resume 为例：
```
prisma/schema.prisma           # model Resume
backend/src/modules/resumes/resumes.service.ts  # 业务逻辑
backend/src/modules/resumes/resumes.controller.ts  # HTTP 接口
backend/src/modules/resumes/dto/  # 请求/响应 DTO
frontend/src/stores/resumeStore.ts  # 状态管理
frontend/src/pages/candidates/CandidateDetail.tsx  # 展示简历详情
```

docker-compose.yml 要求：

- 所有服务使用 `container_name` 命名容器，带 `${COMPOSE_PROJECT_NAME:-ldtalent}` 前缀

- 前端通过 Nginx 反向代理后端 API（调用 `/api`，由 Nginx 转发到 `backend:38502`）
- 数据库配置 `healthcheck`，后端 `depends_on` 数据库健康
- 后端配置 `healthcheck`，前端 `depends_on` 后端健康
- 所有可变配置通过环境变量注入
